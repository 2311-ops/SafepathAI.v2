## Phase Goal

**As a** family member, **I want to** see my family's live and past location on a shared map and control exactly what I share and with whom, **so that** we can keep each other safe without giving up our privacy.

<objective>
Deliver the mobile gateway slice end-to-end: the persistent 5-tab shell, the value-first permission priming screen that unlocks everything (LOC-05, the phase's north-star CTA "Turn on location sharing"), the battery-transparency explainer (LOC-04), and a Live Map that shows the user their own real-time position (LOC-01) using foreground-only geolocator streaming (D-01). After this plan a real user can open the app, understand why location matters, grant permission on their own terms, and see themselves live on the map — with pings flowing to the P02 backend.

Purpose: This is the first phase to build the bottom-nav shell (Phase 1 shipped only a stub) and the first to introduce `google_maps_flutter`/`geolocator`. Getting the permission-priming gate right (never call `requestPermission()` before the priming CTA — Pitfall 6) is the difference between a respectful onboarding and a permanently-lost permission.
Output: shell + router wiring, priming + battery screens, geolocator foreground streaming controller, REST `location_api`, `live_map_screen` with the self pin, and the `MemberMapPin` shared widget.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| OS location permission → app | User-granted permission gates GPS access |
| app → LocationHub.ReportLocation | Client sends its own fixes (server derives identity, P02) |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-02-13 | Information Disclosure | Over-broad location permission (Always/background) requested prematurely | medium | mitigate | Foreground-only manifests (WhenInUse / FINE+COARSE); no background/Always permission strings (D-01) — avoids invasive prompt + store-review scope creep |
| T-02-14 | Repudiation / UX-harm | OS prompt fired before the user understands why (lost permission) | medium | mitigate | `requestPermission()` strictly gated behind the priming-screen CTA (Pitfall 6); widget test enforces it |
| T-02-04 | Spoofing | Client-reported position trusted as identity | high | transfer | Identity is server-derived at the hub (P02 `Context.UserIdentifier`); the client only supplies coordinates for its own authenticated connection |
| T-02-15 | Tampering | Maps API key exposure in the app bundle | low | accept | Key is client-embedded by design; restrict it (Android app signature / iOS bundle id) in Google Cloud Console per the user-setup task |
</threat_model>
