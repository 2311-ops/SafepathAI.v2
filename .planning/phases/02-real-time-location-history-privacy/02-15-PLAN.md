<objective>
Deliver the mobile profile-management slice: extend the existing `profile` feature (data/model/api/controller) in place (do NOT create a parallel module, per D-12/canonical refs) and add a `ProfileScreen` where a user can view their identity (PROFILE-05), edit their display name (PROFILE-04), and upload / replace / remove their profile picture (PROFILE-01/02/03). All Storage access is backend-mediated — the app uploads raw picked bytes to the 02-14 endpoints and never calls `supabase_flutter`'s Storage client (D-13).

Purpose: This is the demonstrable half of the upload/replace/remove capability — after this plan a real user can set their name and photo end-to-end against the running backend.

Output: extended profile model/API/controller, a `ProfileScreen` wired into the app shell + router, the two new pub dependencies (`image_picker`, `cached_network_image`), iOS photo-permission usage string, and controller tests.

MVP slice framing (ROADMAP Phase 2 Success Criterion #6): "As a family member, I want to set my display name and profile picture, so that the family map shows who is who." This plan makes that user-visible and interactive.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| user-picked photo -> mobile app -> backend | The app is a courier of untrusted bytes; it performs no security validation (the backend does, 02-13) |
| signed avatar URL -> on-device cache | Cached under a stable key; the URL itself is a short-lived bearer token |

## STRIDE Threat Register (ASVS L1, block on High)

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-02-13 | Tampering | Client-side "validation" trusted as a boundary | High | mitigate | D-16 — client downscale is a UX/bandwidth optimization only; the app never treats its own compression as a trust boundary, and the backend re-validates/re-encodes every upload (02-13). The app must NOT call `supabase_flutter` Storage directly (D-13) |
| T-02-06 | Information Disclosure | Signed avatar URL persistence in cache | Low | accept | cached_network_image stores under a stable non-secret cacheKey (userId+profileUpdatedAt), and the signed URL decays at its 1h TTL; standard app-sandbox cache, matching the phase's accepted eventual-revocation posture |
| T-02-SC | Tampering | pub.dev installs (image_picker 1.2.3, cached_network_image 3.4.1) | Low | accept | Research Package Legitimacy Audit: both from verified publishers (flutter.dev, baseflow.com), high adoption, OK/Approved — no [SUS]/[SLOP], so no blocking-human legitimacy checkpoint required |
</threat_model>
