<objective>
Build the backend foundation for user profile identity: extend the `User` entity with the three profile fields (D-12), and stand up the two Infrastructure services every avatar endpoint depends on — a backend-mediated Supabase Storage client (D-13/D-14) and a server-side image validator/re-encoder (D-15/D-16). This is the enabling slice for PROFILE-01 (upload), PROFILE-02 (replace), and PROFILE-03 (remove); the HTTP endpoints and signed-URL propagation that consume these services land in 02-14.

Purpose: Everything about "who can see whose avatar" must run through the existing family-membership + SharingPreference server-side gate (D-13). A direct-from-client Supabase Storage call would bypass that gate, so all Storage access is centralized here behind Application-layer interfaces, exactly as `FamilyAuthorizationService : IFamilyAuthorizationService` centralizes authorization.

Output: `User` schema migration applied to the live database, `IProfileImageStorage`/`IProfileImageValidator` with their Infrastructure implementations, DI wiring, and the `Supabase:ServiceRoleKey` config seam.

MVP slice framing (derived from ROADMAP Phase 2 Success Criterion #6): "As a family member, I want to set a profile picture and display name, so that the family map shows who is who." This plan delivers the backend storage/validation half of the upload/replace/remove capability; 02-15 delivers the mobile UI that makes it demonstrable end-to-end.
</objective>

| T-02-03 | Tampering | Avatar upload (polyglot file) | High | mitigate | Task 3 step 4 — full `Image.Load` + `JpegEncoder` re-encode persists only decoded pixel data; trailing/embedded non-image bytes are structurally discarded before reaching Storage |
