<objective>
Expose the profile identity capability over HTTP and SignalR by consuming the storage/validation services from 02-13. Add the three profile command handlers, wire them into `MeController`, extend `GET /me` (PROFILE-05) and `GET /families/{familyId}/live-locations` (PROFILE-06/07) with a server-signed `profileImageUrl`, and add the lightweight `ProfileUpdated` SignalR event (D-17) so name/picture changes propagate in real time without bloating the per-ping `LocationUpdated` payload.

Purpose: PROFILE-07's "Family-Circle-only" visibility is satisfied by REUSING `GetLiveLocationsQuery`'s existing `IFamilyAuthorizationService.RequireMembership` + `ISharingAuthorizationService` double-gate — no new authorization path is introduced (D-13/D-17). The write endpoints derive the acting user exclusively from `ICurrentUserService` (no userId in route/body), making IDOR structurally unreachable.

Output: profile write endpoints, signed-URL read integration on both `/me` and live-locations, and the `ProfileUpdated` broadcast seam. After this plan the backend half of PROFILE-01..07 is complete; 02-15/02-16 render it in the app.

MVP slice framing (ROADMAP Phase 2 Success Criterion #6): completes the backend API surface a user's profile edits and the family map both depend on.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| mobile client -> MeController write endpoints | Untrusted image bytes + display-name string cross here; acting user must be server-derived |
| backend -> family SignalR group | ProfileUpdated fan-out must stay scoped to the caller's own family group |
| backend-issued signed URL -> any holder | A signed URL grants time-boxed read to a private object |

## STRIDE Threat Register (ASVS L1, block on High)

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-02-02 | Tampering / EoP | /me/profile-image + /me/display-name (IDOR) | High | mitigate | Task 2 — endpoints carry no userId in route/body; `CallerUserId` is derived solely from `ICurrentUserService`, so a caller can only mutate their own profile |
| T-02-01 | Information Disclosure | live-locations avatar exposure across families | High | mitigate | Task 3 — `ProfileImageUrl` is signed ONLY when the existing RequireMembership + SharingPreference `canViewLocation` gate passes; non-family/non-shared members get null (PROFILE-07) |
| T-02-05 | Denial of Service | oversized upload body | High | mitigate | Task 2 — `[RequestSizeLimit(6_000_000)]` refuses bodies before buffering, ahead of the validator's 5 MB check |
| T-02-06 | Information Disclosure | signed-URL leakage/replay | Medium | accept | Task 1 — 1-hour TTL bounds exposure; production logging must not log signed-URL query strings (same logging-hygiene posture as the SignalR access_token param). Residual: a signed URL is not instantly revoked if SharingPreference toggles off mid-TTL — it decays at expiry (accepted, matches the phase's existing eventual-revocation stance) |
| T-02-SC | Tampering | package installs | Low | accept | This plan adds no new packages (reuses 02-13's ImageSharp and the in-box SignalR) |
</threat_model>
