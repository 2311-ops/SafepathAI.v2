## Phase Goal

**As a** family member, **I want to** see my family's live and past location on a shared map and control exactly what I share and with whom, **so that** we can keep each other safe without giving up our privacy.

<objective>
Close the 02-VERIFICATION.md LOC-05 failed gap. The priming screen already exists and CTA-gates the OS permission request, but a signed-in user currently reaches `/home` directly, which builds `MainShell` and the Map tab before permission is granted. This plan adds the missing app/router/controller gate and proves that cold signed-in `/home` cannot start the map's location pipeline before permission priming.

Purpose: LOC-05 and D-10 are privacy and trust requirements, not only UI copy. The user must understand and choose location sharing before any foreground stream starts.
Output: a `/home` permission gate, controller-level streaming guard, and route/controller tests covering denied/unknown/granted permission states.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|---|---|
| user consent -> OS location API | The app must not start GPS streaming or prompt the OS before the user sees value-first priming and taps the CTA. |
| route state -> location controller | Route/UI state is convenience only; the controller must independently enforce the granted-permission precondition before crossing into platform GPS and SignalR. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|---|---|---|---|---|---|
| T-02-10-01 | Information Disclosure | `LocationController` / `positionStreamProvider` | high | mitigate | Controller-level guard prevents hub connection, live-location fetch, and Geolocator stream subscription until permission is granted. |
| T-02-10-02 | Tampering | `/home` route bypassing priming | medium | mitigate | `/home` is wrapped in `LocationPermissionGate`; route/gate tests cover denied and unknown permission states. |
| T-02-10-03 | Repudiation | OS permission prompt appears without user intent | medium | mitigate | `requestPermission()` remains callable only from the priming CTA; tests assert no request during cold route gating. |
| T-02-10-04 | Denial of Service | Permission check loops or router redirect loops | low | mitigate | Gate renders a neutral loading scaffold during checks and excludes `/permission-priming` from redirect loops; tests cover granted release to `/home`. |
</threat_model>
