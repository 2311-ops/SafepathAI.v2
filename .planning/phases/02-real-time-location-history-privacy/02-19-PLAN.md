## Phase Goal

**As a** family member, **I want to** see my family's live and past location and control exactly what I share and with whom, **so that** we can stay connected safely without giving up privacy.

<objective>
Close verification blocker CR-01 for PRIV-02 / LOC-02: `GetLiveLocationsQueryHandler` currently computes the location-ping recency contribution to `IsOnline` before applying the LiveLocation sharing gate. A caller denied LiveLocation sharing can therefore learn that the target member pinged recently, even though every sibling location-derived field is already nulled.

Purpose: Restore the "control exactly what I share" promise at the API layer while preserving the locked D-03 dual-signal presence model. Only the ping-derived half of `IsOnline` is gated by `canViewLocation`; `_presence.IsOnline(member.UserId)` remains independent connection presence unless a later product decision changes that model.

Output: One backend regression test pair and one scoped handler fix in `GetLiveLocationsQueryHandler`.
</objective>

## Artifacts This Phase Produces

- New test method: `Handle_DoesNotUseRecentPingForIsOnlineWhenLiveLocationSharingDenied`
- New test method: `Handle_PreservesConnectionPresenceWhenLiveLocationSharingDenied`
- Changed behavior: `GetLiveLocationsQueryHandler` computes `isRecent` only when `canViewLocation` is true.
- Preserved behavior: `MemberLiveLocationDto.IsOnline` still returns true when `IPresenceQuery.IsOnline(member.UserId)` is true, even if LiveLocation sharing is denied.
