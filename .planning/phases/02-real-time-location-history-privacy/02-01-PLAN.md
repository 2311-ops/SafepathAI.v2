## Phase Goal

**As a** family member, **I want to** see my family's live and past location on a shared map and control exactly what I share and with whom, **so that** we can keep each other safe without giving up our privacy.

*(Derived faithfully from ROADMAP Phase 2 Goal + Success Criteria + CONTEXT.md phase boundary. The ROADMAP `**Goal:**` line is a plain sentence, not `As a / I want to / so that` form — flagged to the orchestrator; reformat via `/gsd mvp-phase 02` if a canonical story is wanted.)*

<objective>
Stand up the phase's real-time transport as a thin, proven end-to-end walking skeleton and de-risk the `signalr_netcore` `[SUS]` package (RESEARCH.md Package Legitimacy Audit) BEFORE any feature logic is built on it. Delivers a minimal authenticated `LocationHub` (D-02: SignalR push, not REST polling) plus the mobile client wrapper, verified by a live spike: connect, receive one family-scoped broadcast, force a drop, confirm auto-reconnect + group rejoin.

Purpose: The Dart SignalR client ecosystem is thin — `signalr_netcore` is community-maintained with a documented "not receiving message callback" regression and 56 open issues. A dead end here would sink every later real-time plan (P02, P06, P07), so it must be caught cheaply in Wave 1 (RESEARCH.md "Planner action required").
Output: A working real-time channel (backend hub + mobile client) authenticated by the existing Supabase JWT over WSS/TLS (contributes PRIV-01 transport encryption), gated by family membership, with a human-verified reconnect smoke test.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| mobile client → SignalR hub | Untrusted WebSocket handshake; token arrives as `access_token` query string; client-claimed familyId crosses here |
| npm/pub package registry → build | Third-party `signalr_netcore` code enters the app |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-02-SC | Tampering | pub.dev `signalr_netcore` install | high | mitigate | Blocking-human legitimacy checkpoint (Task 1) before it enters pubspec; verify publisher/repo/version on pub.dev; record fallback pin |
| T-02-03 | Information Disclosure | LocationHub group broadcast | high | mitigate | `OnConnectedAsync` calls `RequireMembership` and only adds the connection to `family:{familyId}` after it passes; broadcasts target the group, never all clients (spike Task 4 step 5 verifies isolation) |
| T-02-04 | Spoofing | LocationHub connection identity | high | mitigate | userId derived from `Context.UserIdentifier` (validated JWT sub claim), never from a client-supplied value |
| T-02-01 | Tampering | familyId query-string param | high | mitigate | familyId is re-checked against the caller's active membership server-side (`RequireMembership`); a guessed familyId the caller isn't in throws HubException |
| T-02-06 | Information Disclosure | `access_token` in query string appearing in access logs | medium | accept | TLS/WSS protects it in transit; production request-logging of `/hubs/location` query strings to be suppressed before deploy (tracked, revisited P05/deploy) |
| T-02-07 | Session | JWT not revalidated mid-connection | low | accept | Documented SignalR behavior; acceptable for family location sharing threat model (RESEARCH Pitfall 5) |
</threat_model>
