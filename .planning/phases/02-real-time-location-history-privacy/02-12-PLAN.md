<objective>
Retrofit the Phase 2 map-rendering layer from `google_maps_flutter` (billed Google Cloud Maps SDK) to `flutter_map` + `latlong2` (OpenStreetMap raster tiles). This is a presentation-layer swap of already-shipped, verified code — the feature set does not change, only the rendering engine. Live location, presence, history, privacy, SignalR transport, `geolocator`, and all backend code are untouched.

Purpose: Remove the Google Maps billing/API-key dependency (per the project direction change recorded in `02-OSM-MIGRATION-IMPACT.md` and the already-updated `.claude/CLAUDE.md` stack table) and give Phase 4 geofencing a map renderer it can draw safe-zone circles on without a Maps SDK key.

Output: `pubspec.yaml` on flutter_map/latlong2; rewritten `live_map_screen.dart` and `route_stats_sheet.dart`; Android + iOS native map-key wiring removed; updated widget tests re-confirming LOC-01/LOC-02/LOC-04/HIST-02 render under the new renderer; `02-01-USER-SETUP.md` reflecting the no-API-key OSM setup path.

This plan preserves the public API of both surfaces exactly (`LiveMapScreen`, `showRouteStatsSheet(...)`, `RouteStatsSheet`) so every existing caller and test binds unchanged. It is additive to a CLOSED, verified phase — plans 02-01..02-11 are historical record and are NOT touched.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| App → OSM tile server (`tile.openstreetmap.org`) | The app fetches raster map tiles over HTTPS from a third-party public server. The requested z/x/y tile coordinates reveal the approximate geographic viewport being displayed (centered on the user's / family's location). |
| Build config → pub.dev | Two new third-party Flutter packages (`flutter_map`, `latlong2`) are added to the dependency graph, replacing `google_maps_flutter`. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-02-M1 | Information Disclosure | `TileLayer` requests to `tile.openstreetmap.org` | medium | accept (dev) / mitigate (prod) | OSM's public tile server sees the approximate viewport (near the family's location), in tension with the privacy-first / no-data-resale posture. Accept at graduation/dev scale; `02-01-USER-SETUP.md` documents moving to a privacy-respecting or self-hosted tile provider before production traffic. No precise coordinates or user identifiers are sent — only tile indices over HTTPS. |
| T-02-M2 | Tampering | OSM tile fetch over the network | low | mitigate | Use the HTTPS tile URL only (`https://tile.openstreetmap.org/{z}/{x}/{y}.png`), never HTTP; tiles are non-security-critical imagery with no code-execution path. ATS/HTTPS prevents on-path tampering of tile content. |
| T-02-SC | Tampering | `flutter pub add flutter_map latlong2` (supply chain) | low | mitigate | `flutter_map` + `latlong2` are the `.claude/CLAUDE.md`-sanctioned, community-standard OSM stack (see 02-OSM-MIGRATION-IMPACT.md §3). Pin to `^8.x` / `^0.9.x`, resolve from pub.dev only, and let `pubspec.lock` record content hashes. Do not add any other map/tile package. (Not an npm/pip/cargo ecosystem, and the packages are pre-vetted in project docs, so no blocking legitimacy checkpoint is required.) |
</threat_model>
