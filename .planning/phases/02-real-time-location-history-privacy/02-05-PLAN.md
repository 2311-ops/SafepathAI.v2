## Phase Goal

**As a** family member, **I want to** see my family's live and past location on a shared map and control exactly what I share and with whom, **so that** we can keep each other safe without giving up our privacy.

<objective>
Close the phase's backend: (1) low-battery alerts (NOTIF-01) delivered in-app via the hub with falling-edge suppression so a phone sitting at 15% does not spam dozens of duplicate alerts; (2) the Privacy Center's data export (JSON of the caller's own records — D-09) and irreversible delete (PRIV-04); and (3) the PRIV-05 no-data-resale commitment surfaced as retrievable policy text. There is no FCM in this phase (that is Phase 3), so the low-battery alert rides the existing SignalR hub as an in-app event.

Purpose: Alert-spam suppression and owner-scoped export/delete are correctness + privacy requirements. Delete must be a genuine hard delete of only the caller's rows (never another user's), matching the UI-SPEC's "This can't be undone" contract.
Output: `LowBatteryEvaluator` + `LowBatteryAlertTracker` wired into `ReportLocationCommandHandler`, a new `LowBattery` client method on the hub contract, `ExportMyDataQuery`/`DeleteMyDataCommand`, PrivacyController export/delete/policy endpoints.
</objective>

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-02-01 | Tampering / Information Disclosure | Export/delete acting on another user's data (IDOR) | high | mitigate | Both scoped to `CallerUserId` from `ICurrentUserService`; queries/deletes filter `WHERE UserId = caller`; no user id accepted from client |
| T-02-11 | Denial of Service | Battery-alert spam flooding family via the hub | medium | mitigate | Falling-edge hysteresis (alert 20 / clear 25) via `LowBatteryEvaluator` + `LowBatteryAlertTracker` — one alert per genuine crossing |
| T-02-02 | Information Disclosure | Battery alert leaking presence of a non-sharing member | medium | mitigate | Alert broadcast reuses the LiveLocation sharing recipient filter (P03) |
| T-02-12 | Repudiation | Irreversible delete with no confirmation | low | accept | Confirmation friction lives in the mobile UI (UI-SPEC destructive-dialog copy, P09); the endpoint itself is intentionally a hard delete |
</threat_model>
