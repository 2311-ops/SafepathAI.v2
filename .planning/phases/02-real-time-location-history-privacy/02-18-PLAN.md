<objective>
Close UAT test 73 (major): the Live Map header avatar reverts to a stale/incorrect photo after fully closing and reopening the app, even though the in-session live-update fix (02-17) works while the app stays open.

Root cause (fully diagnosed in .planning/debug/header-avatar-cold-start-stale.md): the backend `GET /families/{familyId}/live-locations` endpoint — the single source `LocationController._bootstrap()` calls on every cold app start for both `selfPosition` and all family members — never returns `profileUpdatedAt`. `MemberLiveLocationDto` has no such field and `GetLiveLocationsQueryHandler`'s projection never selects `user.ProfileUpdatedAt`. So `LiveLocation.fromJson` always parses `profileUpdatedAt` as null on cold start, the `CachedNetworkImage` cacheKey collapses to the constant `${userId}-null`, and `cached_network_image`'s disk cache (which persists across app restarts) serves the first-ever cached image bytes forever. In-session updates escape this because the SignalR ProfileUpdated handler client-stamps a fresh local timestamp — but that stamp is never persisted, so the next cold start reverts to the stale image. The authoritative `User.ProfileUpdatedAt` already exists, is stamped on every profile mutation, and is already exposed via `/me` — it was simply never carried into this query.

Purpose: Restore correct avatar rendering on cold start for the header AND all family markers (the bug is systemic to both, per the diagnosis), closing the PROFILE-03/05/06 identity guarantees end-to-end.
Output: Backend DTO + query thread a real, sharing-gated ProfileUpdatedAt on the live-locations endpoint; backend and mobile regression tests guard the cold-start cache-bust chain.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| mobile client -> API (/families/{familyId}/live-locations) | Authenticated (Supabase JWT) HTTPS request; server derives caller identity, never trusts a client-supplied user ID |
| API -> DB (EF Core / Supabase Postgres) | Server-owned query; User.ProfileUpdatedAt is authoritative, stamped only by profile-mutation commands |

No new trust boundary is introduced: this change adds one already-authoritative timestamp field to an existing authenticated, sharing-gated endpoint.

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-02-18-01 | Information Disclosure | GetLiveLocationsQueryHandler ProfileUpdatedAt projection | low | mitigate | Gate ProfileUpdatedAt behind the existing canViewLocation sharing check (identical to ProfileImageUrl), so a viewer who cannot see a member's live location learns nothing about when that member last changed their profile. Backend test asserts the gated-null case. |
| T-02-18-02 | Tampering | profileUpdatedAt / client cache key | low | accept | The timestamp is server-sourced from User.ProfileUpdatedAt (authoritative, stamped only by profile-mutation commands) and is read-only on the client; it is never accepted from the client. No integrity control needed. |
| T-02-18-SC | Tampering | package installs (npm/nuget/pip) | n/a | accept | No new dependencies added — pure code change to an existing DTO record and query projection plus tests. No package legitimacy gate applies. |
</threat_model>
