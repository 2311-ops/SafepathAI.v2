## Phase Goal

**As a** family member, **I want to** see my family's live and past location on a shared map and control exactly what I share and with whom, **so that** we can keep each other safe without giving up our privacy.

<objective>
Make live location real end-to-end on the backend: persist raw pings, broadcast them to the family group over the P01 hub (D-02), and serve each member's last-known position for the map's initial load. Presence is computed from BOTH connection state (P01 `PresenceTracker`) and ping recency (D-03), so the UI can render "connected but stale". Accuracy and timestamps pass through unchanged so the mobile staleness/accuracy treatment (LOC-03) has the data it needs.

Purpose: This is the location data spine every later plan reads from (history in P04, privacy gating in P03, mobile map in P06/P07). Persist-then-broadcast keeps the write authoritative and the push a follow-up (never a gate), matching the SOS-priority architecture CLAUDE.md mandates.
Output: `LocationPing` entity + migration, `ReportLocationCommand` (persist + broadcast), `GetLiveLocationsQuery`, a `LocationController` GET surface, and centralized `GeoMath`/`DwellTimeDefaults` for Phase 4 reuse.
</objective>

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-02-01 | Tampering / Information Disclosure | GET live-locations (IDOR via guessed familyId) | high | mitigate | `RequireMembership(CallerUserId, FamilyId)` runs before any data is read; caller id from `ICurrentUserService`, familyId re-checked against caller's own membership |
| T-02-04 | Spoofing / Tampering | ReportLocation payload (spoofed UserId or coordinates) | high | mitigate | UserId derived from `Context.UserIdentifier`; lat/lng bounds + not-future timestamp validated; invalid payloads rejected and not persisted |
| T-02-08 | Tampering | Out-of-range coordinate / battery injection polluting history | medium | mitigate | FluentValidation-style bounds check in the handler before persistence (V5 Input Validation) |
| T-02-03 | Information Disclosure | Broadcast fan-out to wrong recipients | high | mitigate | Broadcast targets the caller's `family:{familyId}` group only; per-recipient gate added in P03 (SharingPreference) |
</threat_model>
