# Phase 01.1 — UI Design Contract

> Visual and interaction contract for the animated logo splash screen (SafePath AI Flutter app, cold launch only). Generated by gsd-ui-researcher, verified by gsd-ui-checker.

The authenticated fast-path (splash → Home, `appBg` light background) will still show a color shift at the handoff; this is unavoidable without inventing a new background, and should be smoothed by the exit fade (see Motion Contract).

## Motion & Animation Contract

**Baseline (REQUIRED).** Single `AnimationController`, no `Timer`/`Future.delayed` orchestration, calm curves only (`Curves.easeOutCubic` / `Curves.easeOutQuart` / `Curves.fastOutSlowIn`). No bounce, elastic, overshoot, large rotation, flashing, strong/pulsing glow, particles, shaders, or looping.

| Property | Value |
|---|---|
| Total duration | **1400ms** default (tunable 1200–1600ms by planner/executor; never outside this window) |
| Background | appears **immediately** — set as a static `Scaffold`/`Container` gradient property (not animated, not faded in). This guarantees no blank/black/white frame is ever visible, including frame 0. |
| Entry interval | 0%–~55% of total duration (~770ms @ 1400ms), `Curves.easeOutQuart` |
| Entry motion | The logo lockup (mark + wordmark treated as **one** child, see below) animates together: opacity 0→1, scale 0.92→1.0, vertical rise 8px→0px (within the locked 6–8px range) |
| Hold interval | ~55%–85% of total duration (~420ms @ 1400ms) — content at rest, no additional motion |
| Reserved tail | ~85%–100% (~210ms @ 1400ms) — this is where the completion gate (see Integration Contract) is satisfied and navigation fires; no new visual motion is required here — see "Exit" below |
| Exit / transition out | Navigation fires once `animationComplete && initializationComplete`. Prefer a short (~150–200ms) fade-in `CustomTransitionPage`/`FadeTransition` on the **destination** route so the handoff reads as one continuous motion rather than a jump-cut — do not build a second animated "exit" sequence inside the splash widget itself (keeps the controller and its motion simple, avoids a second timing budget to coordinate) |
| Logo lockup composition | `SafePathLogo(size: 96, tile: false)` + `SizedBox(height: AppSpacing.lg)` + `Text('SafePath AI', style: AppTypography.display.copyWith(color: Colors.white))`, wrapped in one `Column(mainAxisSize: MainAxisSize.min)` — **identical sizing/spacing/style to `WelcomeScreen`**, animated as a single fade+scale+rise unit (not per-element, not per-letter) |
| Reduced motion | If `MediaQuery.of(context).disableAnimations` (or the equivalent `SchedulerBinding` accessibility check) is true: skip scale and rise entirely; use opacity-only fade 0→1 over ~200–250ms, single `Curves.easeOutCubic`. The full lockup (mark + wordmark) must still fully appear — never a blank or partial-content hold. The completion gate logic is unchanged (still waits for init). |
| Disposal | `AnimationController` and any `addStatusListener` must be removed/disposed in `dispose()` |
| Repaint isolation | Wrap the animated logo lockup subtree in a `RepaintBoundary` so the `Scaffold`/`SafeArea` never repaint per frame |

**Optional flourishes.** May be added only if every constraint below is met. If a flourish cannot meet its constraint, omit it — the baseline above is a complete, shippable splash on its own.

| Flourish | Allowed if… | Flag / required deviation from the reference `splash_screen.dart` |
|---|---|---|
| Slow rotating halo behind the mark | Constant, non-pulsing opacity ≤0.15 using only `#5FD0C5`; rotation is a slow continuous drift (≤0.15 full turns over the entire duration) — never accelerates, never blinks | **FLAGGED.** The reference implementation's `glowOpacity = 0.14 + 0.07 * sin(t * π * 1.4)` is a continuously *oscillating* ("breathing") opacity — this is a low-grade pulse and risks reading as flashing under the "no flashing" accessibility constraint. **Required deviation:** drop the `sin()`-based breathing term; use a flat, constant low-opacity halo (e.g. fixed `0.15`) if this flourish is implemented at all. |
| One-time diagonal sheen sweep across the mark | Plays exactly once, within the entry/hold window, ≤0.25 opacity, white or `#5FD0C5` only, `easeOutCubic`, never repeats | Reference implementation is compliant as-is on this one — safe to reuse its `_SheenSweep` approach if desired, just rescale its `_start`/`_end` fractions to the 1400ms budget instead of 1800ms. |
| Staggered per-letter wordmark reveal | Each letter uses `easeOutQuart`, ≤30ms stagger, and **all letters must finish within the controller's own 0–100% range** (not overflow past `AnimationStatus.completed`) | **FLAGGED.** The reference's `_wordStart = 0.55` + 10×`28ms` stagger + `220ms` per-letter duration finishes at ≈105% of the controller's duration — letters are still animating in *after* the controller reports `completed`, which breaks the animation-complete gate this whole contract depends on (navigation could fire mid-letter-reveal). **Required deviation:** if implemented, recompute `wordStart` so `wordStart + (letterCount × stagger) + letterDuration ≤ 1.0`, or reserve a dedicated sub-interval for it inside the existing entry window rather than starting mid-hold. |
| Total duration | must be 1200–1600ms | **FLAGGED.** Reference implementation uses **1800ms** total (900 entry + 500 hold + 400 reserved-tail), which exceeds the locked 1200–1600ms brief. **Required deviation:** rescale to the 1400ms default above (or any value in 1200–1600ms), keeping the same proportional split if flourishes are kept. |

**Reference implementation navigation pattern — do not reuse.** `splash_screen.dart`'s `AppRouter.goToHome(context)` / `AppRouter.goToWelcome(context)` static-method calls do not exist against this app's real router (`mobile/lib/core/router/app_router.dart` exposes a Riverpod `routerProvider` with a single declarative `redirect` callback, not imperative navigation helpers). Do not add `AppRouter.goToHome`/`goToWelcome` methods to match the reference — instead follow the Integration & Navigation Contract below, which reuses the existing redirect system per the locked technical decision ("prefer existing redirect system over manual pushes").

**Reference implementation import paths — do not reuse verbatim.** The reference file imports `shared/widgets/safepath_logo.dart`; the real path is `mobile/lib/shared_widgets/safepath_logo.dart` (no `shared/widgets/` nesting). Adjust on drop-in.

## Integration & Navigation Contract

- **New route:** add `/splash` to `routerProvider` (`mobile/lib/core/router/app_router.dart`) and set it as `initialLocation` (replacing `/` as the *first* route only — `/` / `WelcomeScreen` remains fully intact and reachable exactly as today).
- **Completion gate:** navigate away from `/splash` only when **both** `animationComplete` and `initializationComplete` are true (locked decision). Today, `AuthController.build()` resolves synchronously — `Supabase.initialize()` is already `await`ed in `main()` before `runApp`, so session restoration is effectively instant by the time the widget tree builds. This means, in practice, `initializationComplete` is already true on first frame; the gate exists so (a) the animation still plays its full minimum duration rather than being truncated by an instant init, and (b) the architecture is ready for any future async init step without redesign.
- **Recommended mechanism (Claude's Discretion — reuses the existing redirect system, no manual `context.go`/`push` from inside the splash widget):**
  1. Add a small Riverpod state, e.g. `splashAnimationCompleteProvider` (`StateProvider<bool>`, default `false`).
  2. `SplashScreen` flips it to `true` exactly once — either on `AnimationStatus.completed`, or immediately if reduced motion is active — guarded the same way the reference implementation guards `_navigated` (a local bool checked before the state write, so rebuilds/lifecycle resumes can't double-fire).
  3. Extend the existing `redirect` callback in `routerProvider`: while `state.matchedLocation == '/splash'` and `!ref.read(splashAnimationCompleteProvider)`, return `null` (stay put — this is what prevents the redirect logic from instantly bouncing off `/splash` before the animation has even started, since auth state is already known synchronously). Once the flag flips true, fall through unchanged to the existing redirect rules (which already correctly send authenticated users to `/home` and unauthenticated users to `/`).
  4. Do **not** add `/splash` to `_authenticatedOnlyRoutes` or `_unauthenticatedOnlyRoutes` — it is gated purely by the new boolean check, evaluated before the existing auth checks.
  5. Because go_router redirects *replace* rather than push, `/splash` is never left on the back stack — a user can never navigate back into it.
- **Init failure:** if session/family/profile checks fail, this phase does nothing new — the existing redirect rules already send an unauthenticated user to `/` where existing error handling (`AuthError` states, retry flows) is untouched. The splash's only job is to not block that path indefinitely (bounded by the completion-gate + the existing app's own timeout/error behavior, neither of which this phase modifies).

## Platform / Responsive Contract

- Targets: Flutter Web, Android, iOS; phone and tablet; portrait and landscape.
- Layout: `Scaffold` → gradient background → `SafeArea` → `Center` → `RepaintBoundary` → the fade/scale/rise-wrapped logo lockup `Column(mainAxisSize: MainAxisSize.min)`. No `Spacer`/flex-ratio layout is needed (unlike `WelcomeScreen`, which balances bottom CTAs) — the lockup is small and fixed-size (~96px mark + 24px gap + one line of 38px text ≈ 166px tall), so `Center` alone avoids overflow on any realistic viewport including small landscape phones.
- No layout shift: the lockup's box size does not change during the scale animation's visual bounds in a way that reflows siblings (there are none) — `Transform`/`FadeTransition`/`ScaleTransition` are compositor-only and never trigger relayout.
- No overflow: if a future device audit finds a viewport shorter than the lockup's fixed height (e.g. certain foldable cover screens), fall back to wrapping in `SingleChildScrollView` with `Center` inside rather than shrinking the logo — do not introduce a `FittedBox` scale-down by default (keeps sizing exactly matched to `WelcomeScreen`, avoiding an inconsistent logo size between the two adjacent screens).

## Accessibility Contract

- Reduced motion: see Motion & Animation Contract above (opacity-only, ~200–250ms, full content still shown, same completion gate).
- No flashing: satisfied by construction — static background, single monotonic entry fade/scale/rise, at most one constant (non-oscillating) low-opacity halo and/or one single-play sheen if flourishes are used (see flags above).
- Contrast: white wordmark (`AppTypography.display.copyWith(color: Colors.white)`) on the `heroGradientStart`→`deepTeal` gradient is reused verbatim from the already-shipped `WelcomeScreen` — no new contrast check is needed, it is provably already acceptable in production.
- Semantics: the `SafePathLogo` `CustomPaint` mark is purely decorative and must be excluded from the semantics tree (e.g. wrap in `ExcludeSemantics`, or ensure the `CustomPaint` has no implicit semantic label). The "SafePath AI" `Text` widget already provides a real, screen-reader-readable accessible name — do not add a redundant separate `Semantics(label: 'SafePath AI')` wrapper unless testing shows the two children get announced as separate awkward nodes, in which case wrap the whole lockup in one `Semantics(label: 'SafePath AI', container: true)` and exclude both children individually.
- Light/dark: **FLAGGED — no dark theme exists in this app today.** `buildSafePathTheme()` hardcodes `Brightness.light`; the reference `splash_screen.dart` branches on `Theme.of(context).brightness` and defines its own unwired dark literal (`#0C3A3F`) that has no corresponding app-level dark `ThemeData`. Recommendation (Claude's Discretion, not locked by any CONTEXT.md): do **not** branch on `Theme.of(context).brightness` or `MediaQuery.platformBrightness` in this phase — always render the single existing light gradient, matching every other screen in the app today. Introducing real dark-mode support is out of scope for this additive phase; branching the splash alone would create a screen that looks different from every other screen depending on system setting, which is a worse outcome than "always light" until a dedicated dark-mode phase exists.

## States

| State | Behavior |
|---|---|
| Fast init (common case today) | Animation still plays its full minimum duration (1200–1600ms) — a fast/instant init never truncates it |
| Slow init (future-proofing) | Completed animation holds its final at-rest frame (no loop, no blank frame) until `initializationComplete` becomes true |
| Init failure | No error UI on the splash itself — falls through to the existing unauthenticated redirect and its existing error handling, bounded (never blocks indefinitely) |
| Reduced motion | Short opacity-only fade, same completion gate, same eventual destination |
| Repeat cold launches | Plays exactly once per cold launch — the `/splash` route is only ever the `initialLocation`, never re-pushed, never a back-stack destination |

## Testing Contract

Deterministic, no network — mirrors the locked decision's coverage list, for the planner to turn into concrete test tasks:

- Animation runs exactly once on `initState`/mount; controller is never re-forwarded.
- Navigation fires exactly once, only when both `animationComplete` and `initializationComplete` are true — assert no double-navigation from rebuilds, provider refreshes, widget lifecycle resumes, or duplicate listeners.
- Reduced motion path: verify the short opacity-only fade runs and the same completion gate/destination logic applies.
- Disposal: `AnimationController` (and any status listener) is disposed with no leaks/no "used after dispose" errors.
- Fast-init case: full minimum duration (1200–1600ms) still elapses before navigation.
- Slow-init case (simulate a delayed `Future`): completed animation holds at rest, no blank frame, no loop, navigates once init resolves.
- Existing auth/role routing is unchanged: authenticated users still land on `/home`, unauthenticated on `/`, recovery sessions still redirect to `/reset-password`, pending-invite redirect still works — all via the existing, untouched redirect rules, now simply gated behind the new `/splash` boolean check.
- `flutter analyze` and the full existing `flutter test` suite must continue to pass with zero regressions.
