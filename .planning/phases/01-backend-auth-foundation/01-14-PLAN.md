<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| OS deep link (safepathai://) -> app | Untrusted external input; the app routes it but never trusts its contents as authorization |
| Invite link token -> backend redeem | The token is validated server-side (expiry, single-use, status) on redeem; the client only forwards it |
| Supabase recovery/oauth deep links -> supabase_flutter | Must remain handled by Supabase; the app listener must not swallow them |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-14-01 | Spoofing / Tampering | A crafted safepathai://invite link carrying a forged/guessed token | high | mitigate | The client only opens the prefilled Accept screen; redemption is validated entirely server-side (expiry, single-use, status, rate-limited) by the existing RedeemInviteCommand — a bad token simply fails with a 400 the amber banner shows |
| T-14-02 | Information Disclosure / DoS | The new listener intercepting/swallowing Supabase's success recovery or OAuth deep links | high | mitigate | The listener acts only on host=invite and on reset-password links carrying an explicit error fragment; all other safepathai URIs are ignored so supabase_flutter continues to process them |
| T-14-03 | Elevation of Privilege | A deep link auto-joining a family without user consent | high | mitigate | The deep link never calls redeemInvite; it only prefills the Accept screen, and joining requires an explicit Accept & join tap |
| T-14-04 | Information Disclosure | Diluting SOS red by using it for routine auth errors | low | mitigate | Reset error surfaces use caution amber only; no red-family color remains on these screens (DESIGN-01 red-reservation) |
</threat_model>
