<objective>
Close a real gap found during Phase 1 manual UAT (this session): 01-07 shipped a working
family-circle UI, but the backend (01-05) never had a "list my families" endpoint (out of that
plan's frozen scope), so the mobile app holds the current family/member list purely in Riverpod
memory for the session that created or joined it. Logging out and back in â€” a completely normal
user action â€” silently loses that state, leaving the Guardian's permissions/invite/landing
screens looking empty even though the family and its members still exist server-side.

Purpose: FAM-01/04/05 need to actually work across a realistic login lifecycle, not just within
one continuous app session.
Output: `GET /families/mine` returns the caller's active family memberships; the mobile app calls
it on auth becoming `Authenticated` (fresh login AND cold app start) and populates
`FamilyController`'s state from it, so the family circle is never "lost."
</objective>

<locked_decisions_applied>
- **D-10-1 (single-family assumption, matches existing schema):** Nothing in the current schema
  (`FamilyMember.FamilyId`/`UserId`) prevents a user from having multiple active memberships, but
  every existing screen/flow (create-circle redirects straight to `/home`, no family-switcher UI
  anywhere) assumes exactly one. `ListMyFamiliesQuery` returns **all** active memberships (don't
  artificially cap it server-side â€” that would be a real data-integrity foot-gun later), but the
  mobile `FamilyController` takes the **first** one for now, with a `// TODO(multi-family):` code
  comment marking the simplification â€” do not build a family-switcher UI in this plan, that is
  out of scope.
- **D-10-2 (endpoint shape, mirrors ListFamilyMembersQuery's existing pattern):**
  `GET /families/mine` -> `IReadOnlyList<MyFamilyDto>` where
  `MyFamilyDto(Guid FamilyId, string FamilyName, Role Role, PermissionLevel Permissions)`. No
  `familyId` route param (unlike the existing member-scoped endpoints) since this is
  inherently "whatever families I belong to," not membership-gated by a specific family â€” the
  `ICurrentUserService.UserId` claim is the only input, same auth pattern as every other endpoint
  in this controller.
- **D-10-3 (fetch trigger, mobile):** `FamilyController` fetches on two triggers: (a) its own
  `build()` if `ref.read(authControllerProvider) is AuthAuthenticated` at construction time
  (covers cold app start with an existing session), and (b) via `ref.listen` on
  `authControllerProvider` transitioning into `AuthAuthenticated` from any other state (covers a
  fresh login). Do NOT poll or refetch on every screen visit â€” this is a one-time-per-session
  bootstrap, matching the existing `AuthController` pattern of reacting to auth state transitions
  rather than screen lifecycle.
- **D-10-4 (no member-name gap fix here â€” separate, larger scope):** The other known 01-07 gap
  (`FamilyMemberDto` has no `fullName`, member rows show raw IDs) is explicitly OUT of scope for
  this plan â€” it needs a `Users` table join the existing `ListFamilyMembersQuery` doesn't do, and
  is a display-polish issue, not a data-loss issue. Do not expand scope to fix it here; note it in
  the SUMMARY as still-open if it comes up again.
</locked_decisions_applied>
