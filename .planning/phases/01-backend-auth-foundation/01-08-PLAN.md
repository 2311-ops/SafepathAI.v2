<locked_decisions_applied>
- **D-08-1 (flow choice):** Use `supabase_flutter`'s `signInWithOAuth(OAuthProvider.google)` —
  Supabase's own "native" OAuth flow (external browser/Custom Tab + PKCE + deep-link redirect
  back into the app). Do NOT add the `google_sign_in` package or the native-ID-token flow
  (`signInWithIdToken`) — that would need the Android "installed"/iOS client IDs, SHA-1
  fingerprint registration, and a `serverClientId`, none of which are set up, and the user's
  brief explicitly says "Supabase Auth's native Google OAuth flow." `signInWithOAuth` needs
  only the Web OAuth client already configured in the Supabase dashboard (per the user: "Google
  OAuth Provider is already configured in Supabase" — do not touch that config or its secret).
- **D-08-2 (redirect URL reuse):** Reuse the existing `supabaseRedirectUrl` constant
  (`safepathai://reset-password`, `mobile/lib/core/config/supabase_config.dart`) as the OAuth
  `redirectTo` too, instead of registering a second redirect URL in Supabase's dashboard. Supabase's
  allow-list matches the exact redirect URL string, not the auth flow it's used for — since this
  URL is already allow-listed (password reset already sends users through it), reusing it means
  **zero Supabase dashboard changes**, honoring "do not modify existing Supabase project
  configuration." The existing `AuthController` listener already treats any `session != null`
  as `AuthAuthenticated` regardless of which flow produced it, and OAuth sign-in fires
  `AuthChangeEvent.signedIn` (not `.passwordRecovery`), so no `AuthRecovery` misrouting occurs.
- **D-08-3 (deep-link platform registration — prerequisite gap, not scope creep):** Neither
  `AndroidManifest.xml` nor `ios/Runner/Info.plist` currently register `safepathai://` as a
  handled URL scheme (`grep` confirms no `intent-filter`/`CFBundleURLTypes` entry exists today).
  This means the *existing* password-reset deep link cannot open the app either — it silently
  falls through to a browser "can't open link" page (reproduced live during phase-1 UAT). Google
  OAuth's redirect-back needs this exact same registration, so this plan adds it once and fixes
  both flows. This is a pre-existing gap being closed as a prerequisite, not new scope.
- **D-08-4 (role assignment for Google sign-ups):** Do NOT build a post-OAuth role-selection
  screen or a new "set role" backend endpoint. The existing `handle_new_auth_user` Postgres
  trigger (`20260708234008_SyncSupabaseUsersAndDropLegacyAuthColumns.cs`) already does
  `COALESCE(raw_user_meta_data->>'role', 'Member')` — a Google sign-up (whose metadata has no
  custom `role` key, only Google profile fields) lands as `Member` automatically, with zero new
  code. AUTH-05 ("assigned a role during setup") is satisfied by this sensible default for the
  Google path; explicit role choice remains exclusive to the email/password Register flow, where
  it is a meaningful choice (Guardian vs Member vs Caregiver vs OrgAdmin).
- **D-08-5 (Register screen — deliberately excluded):** Do NOT add the Google button to
  `RegisterScreen`. Register is step 1 of a two-step drafted flow (`RegisterDraft` ->
  `RoleSelectScreen` -> `register()`) that only makes sense for password-based sign-up; Google
  auth is a single atomic action with no draft step. Welcome already serves as the "new user"
  entry point and carries the Google button, so a second copy on Register would be a redundant,
  confusing mid-form entry point. (This is the plan's use of the brief's own "if appropriate"
  latitude — document, don't silently skip.)
- **D-08-6 (mid-flow cancellation recovery):** `signInWithOAuth`'s returned `Future<bool>`
  resolves once the external browser/Custom Tab is *launched*, not once the user finishes in it
  — so a user who opens the Google picker and then backs out produces no exception and no auth
  event, leaving the controller stuck in `AuthLoading` forever if nothing else intervenes. Fix:
  `AuthController` registers a `WidgetsBindingObserver` (added in `build()`, removed via
  `ref.onDispose`) that, on `AppLifecycleState.resumed`, resets `state` to `AuthUnauthenticated`
  **only if** `state is AuthLoading` AND `authApi.currentSession == null`. This is a deliberate
  heuristic (documented here, not hidden): by the time the app backgrounds-then-resumes, any
  in-flight email/password `login()`/`register()` call has already resolved synchronously, so the
  heuristic cannot misfire on those paths. Unit-test the observer's callback directly (call it as
  a plain function against a fake lifecycle state) rather than driving real OS lifecycle events.
</locked_decisions_applied>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| App -> Google (via Supabase-brokered OAuth) | External browser/Custom Tab redirect; app never sees the Google password, only the final Supabase session |
| Google -> App (deep-link redirect) | `safepathai://reset-password` callback carries the OAuth result; now handled identically for both password-recovery and OAuth sign-in |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-08-01 | Spoofing | A malicious app registering the same `safepathai://` scheme could intercept the OAuth redirect on Android (scheme hijacking — well-known custom-scheme weakness vs. Android App Links) | low | accept | Supabase's PKCE flow means an intercepted redirect alone cannot complete the token exchange without the original code verifier held in the legitimate app's memory; full mitigation (Android App Links / verified `https` deep links) is out of scope for this MVP plan — documented as accepted residual risk, consistent with the existing custom-scheme pattern already used for password reset |
| T-08-02 | Information Disclosure | OAuth tokens/state params logged during the redirect | low | mitigate | `supabase_flutter` handles the deep-link token exchange internally; this plan adds no custom logging of the redirect URI or tokens |
| T-08-03 | Denial of Service | Stuck `AuthLoading` state after a cancelled/failed OAuth attempt blocks the user from retrying | medium | mitigate | D-08-6's lifecycle-resume recovery (Task 1) |
</threat_model>
