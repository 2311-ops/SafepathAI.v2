- **D-09-1 (supersedes D-08-1):** Use `google_sign_in` (native package) +
  `Supabase.instance.client.auth.signInWithIdToken(provider: OAuthProvider.google, idToken:
  ..., accessToken: ...)` instead of `signInWithOAuth`. The user explicitly chose this over
  leaving the URL-flash as-is or getting a custom Supabase Auth domain (both offered, both
  declined) — the trade-off (adds a package + native OAuth client dependency) is accepted.
- **D-09-2 (Android OAuth client — user-confirmed, not verified by this plan):** The user has
  confirmed an Android-type OAuth client already exists in Google Cloud project
  `safepathai-501908`, registered with package name `com.safepath.mobile` and the debug
  keystore's SHA-1 (`BA:3B:AA:E7:71:FB:3B:D7:6A:30:9D:F4:EA:6C:46:67:AC:6B:43:39`). This plan
  does not re-verify that registration (no Google Cloud Console access) — if sign-in fails with
  a `PlatformException`/`ApiException: 10` (`DEVELOPER_ERROR`) during manual verification, that
  is the most likely cause and should route back to the user to re-check Cloud Console, not be
  treated as a code bug in this plan's implementation.
- **D-09-3 (serverClientId, not an Android client_id, in code):** `GoogleSignIn` needs a
  `serverClientId` — this is the **Web** OAuth client's `client_id` (the one already configured
  as Supabase's Google provider,
  `40714411115-eb95t45n7o54b9m0rmgkg2j31r65ah37.apps.googleusercontent.com` — public, safe to
  embed in client code same as `SUPABASE_ANON_KEY`), NOT the Android client's ID. This is
  required so Google issues an ID token whose audience Supabase's backend can verify against the
  provider config already in the dashboard (per D-08-2, untouched). Pass it via a new
  `String.fromEnvironment('GOOGLE_SERVER_CLIENT_ID', ...)` constant in `supabase_config.dart`
  (same pattern as `supabaseUrl`/`supabaseAnonKey`) — add it to `mobile/env.json` (gitignored,
  already exists from earlier Supabase setup), do not hardcode it inline.
- **D-09-4 (cancellation handling simplifies — remove D-08-6's hack):** Unlike
  `signInWithOAuth`'s fire-and-forget browser launch, `google_sign_in`'s sign-in call is
  synchronously awaitable end-to-end — cancellation is a normal return value/exception the
  `await` resolves with directly. This means `AuthController`'s `WidgetsBindingObserver` +
  `recoverFromStuckLoadingOnResume` lifecycle-resume hack from 01-08 (D-08-6) is **dead code**
  after this plan and MUST be removed, along with its dedicated tests — verify the exact current
  cancellation signal shape for the resolved `google_sign_in` package version (via
  `pub.dev`/`context7` — the API has changed across major versions, e.g. `signIn()` returning
  `null` in older APIs vs a typed exception in `GoogleSignIn.instance.authenticate()` in 7.x+;
  do not assume without checking the actually-resolved version in `pubspec.lock` after `flutter
  pub add`) and map it to the existing `AuthUnauthenticated` (no error) return path — same
  observable behavior as 01-08's `launched == false` case.
- **D-09-5 (role assignment — unchanged from D-08-4):** Native ID-token sign-in still carries no
  custom `role` claim; the existing `handle_new_auth_user` trigger's `COALESCE(...,'Member')`
  default still applies unchanged. No new code needed here.
- **D-09-6 (redirect URL / deep-link registration — unaffected, do not remove):** 01-08's
  Android/iOS `safepathai://` deep-link registration stays — it is still required for the
  password-reset flow (unrelated to this plan) and costs nothing to leave in place.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| App -> Google Play Services (native picker) | On-device, no network round-trip to a browser; Google Play Services itself is the trust boundary now instead of an external browser |
| App -> Supabase (`signInWithIdToken`) | The raw Google ID token is sent directly to Supabase for verification — same trust level as the previous flow's session exchange, just one hop shorter |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-09-01 | Spoofing | A misconfigured/wrong Android OAuth client (mismatched SHA-1) could cause Google to reject or, worse, silently mis-issue tokens | low | accept | Google's own `DEVELOPER_ERROR`/`ApiException: 10` fails loudly rather than silently on SHA-1 mismatch (per D-09-2) — not a silent security gap |
| T-09-02 | Information Disclosure | `GOOGLE_SERVER_CLIENT_ID` is a public identifier (like the anon key), safe to embed | none | accept | Not a secret by Google's own design (server client IDs are meant to be embedded in native apps) |
</threat_model>
