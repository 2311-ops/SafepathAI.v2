<locked_decisions_applied>
- D3: Invites are **share-code/QR only** — no required email field. `FamilyInvitation` uses the RESEARCH Pattern 3 schema: `Code` (short human-readable, e.g. `SP-4K9X`), `InviteeLabel` (nullable, guardian-entered display name), `InviteeEmail` (nullable, unused this phase), `ExpiresAt` (24h), `Status`, single-use, `AcceptedByUserId`. Per RESEARCH Pitfall 4 the QR/link payload carries a **longer opaque `LinkToken`** distinct from the short display `Code`; the redeem endpoint is rate-limited.
- D5: Authorization is **backend Application-layer** via `FamilyAuthorizationService` — NOT Postgres RLS. No `CREATE POLICY` SQL. Every family-scoped handler re-checks the caller's own `FamilyMember` row + role.
- D4: New EF migration generated with 9.x tooling; no 10.x pins.
- `Role`, `PermissionLevel`, `InvitationStatus` enums already exist from plan 01 — reuse them, do not redefine.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Member/Guardian client -> /families/* & /invites/* | Client supplies familyId/memberId/invite code — all untrusted |
| Anyone with a code/link -> /invites/redeem | Bearer-style invite code crosses the boundary |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-05-01 | Elevation of Privilege / Tampering | Broken access control across family boundaries (IDOR) | high | mitigate | `FamilyAuthorizationService` re-verifies the caller's own active FamilyMember row + role on EVERY family-scoped handler; cross-family + removed-member denial proven by integration tests (Tasks 1,3). Backend-layer auth is the primary mechanism (D5), not RLS |
| T-05-02 | Spoofing | Invite-code brute-forcing to join a circle uninvited | high | mitigate | CSPRNG short code + longer 32-byte opaque link token, 24h expiry, single-use, redeem requires authentication, rate-limited `/invites/redeem` (Task 2, Pitfall 4) |
| T-05-03 | Elevation of Privilege | Non-Guardian generating invites or changing permissions/removing members | high | mitigate | `RequireRole(..., Role.Guardian)` gate on GenerateInvite / UpdatePermissions / RemoveMember (Tasks 2,3) |
| T-05-04 | Tampering | Orphaning a circle by removing the last Guardian | low | mitigate | Remove-member guards against removing the last active Guardian (Task 3) |
| T-05-SC | Tampering | NuGet supply chain (no new external packages) | low | accept | No new third-party packages introduced; reuses plan-01 dependencies already VERIFIED in RESEARCH audit |
</threat_model>
