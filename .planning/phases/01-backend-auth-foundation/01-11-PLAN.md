<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Supabase-issued JWT -> backend resource server | The backend trusts only issuer/audience/lifetime-validated Supabase tokens; it mints nothing |
| Developer machine / CI host -> backend config | Secrets enter via `.env` (local) or host-injected env vars (deploy); never via tracked files |
| Client-supplied role claim -> authorization | A JWT `role` claim is untrusted for domain authorization; domain role comes from the DB |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-11-01 | Information Disclosure | Secrets in `.env` / `client_secret_*.json` | high | mitigate | Keep `.gitignore` entries for `.env`, `*.env`, `client_secret_*.json`; ship only `.env.example` with placeholders; acceptance criterion asserts `git ls-files` tracks no real secret file. Verified never committed (git log empty) |
| T-11-02 | Spoofing / Elevation of Privilege | Trusting a forgeable `role` claim in the JWT for domain authorization | high | mitigate | Domain role resolved server-side from the Users table via GetMeQuery; the JWT `role` claim (stock value `authenticated`) is never used to grant Guardian/Member privileges. Family-scoped authorization continues to re-check the FamilyMember row server-side (existing FamilyAuthorizationService) |
| T-11-03 | Denial of Service | Backend fails to boot when `DefaultConnection` is blank and `.env` is unwired | medium | mitigate | Env.Load wires `.env` at startup; `.env.example` documents required keys so a fresh clone boots deterministically |
| T-11-04 | Tampering | Supply-chain risk of the added DotNetEnv NuGet dependency | low | accept | DotNetEnv is a widely-used MIT dotenv loader on nuget.org; pinned to an explicit version. NuGet is outside the npm/pip/cargo package-legitimacy gate scope; loader runs only at startup and reads local files |
</threat_model>
