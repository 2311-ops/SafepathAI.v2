<review_findings_addressed>
- HIGH "Single-family invariant missing" (policy half): CreateFamilyCommand and RedeemInviteCommand both add memberships without checking existing active memberships, so multi-family is possible. The whole read path (ListMyFamiliesQuery + mobile "take first") already assumes single-family. Decision (Claude's discretion — no CONTEXT.md locked decision exists): enforce SINGLE active family per user for v1, matching PROJECT.md's "one continuous safety graph" framing and the mobile TODO(multi-family) marker. Enforced server-side at both membership-creating write paths.
- MEDIUM "Rejoin-same-family-after-removal violates the unique (FamilyId, UserId) index" (review finding #6): `RemoveMemberCommand` soft-removes (sets IsActive=false, keeps the row), and `FamilyMemberConfiguration` has a unique index on (FamilyId, UserId). The current `RedeemInviteCommand` Accept branch always inserts a NEW FamilyMember row, so a previously-removed member accepting a fresh invite back into the SAME family would violate that unique index → unhandled DbUpdateException → HTTP 500, with the invitation already flipped to Accepted (consumed) in the failed transaction. Fixed inside Task 1: the Accept branch reactivates the existing (FamilyId, UserId) row in place instead of inserting when one exists. The single-family guard (checking for any ACTIVE membership) still runs first, so this only ever revives an inactive row.
- HIGH "transfer ownership and delete family are absent" — the transfer/delete half is handled in 01-13 (it shares FamiliesController + the DI file with this plan, so it is sequenced into the next wave). Invite REVOKE is added here.
</review_findings_addressed>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Invitee/Guardian app -> /families & /invites | Every membership-creating or invite-lifecycle call is re-authorized server-side |
| Client-supplied familyId / invitationId | Never trusted; handlers re-scope target rows to the authorized family |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-12-01 | Elevation of Privilege | A user joining multiple families to widen their visibility/data surface | high | mitigate | Single active membership enforced server-side at both CreateFamilyCommand and RedeemInviteCommand (Accept), returning 409; a second join is impossible even with a crafted client |
| T-12-02 | Elevation of Privilege | A non-Guardian revoking another member's invite | high | mitigate | RevokeInviteCommand calls RequireRole(Guardian); a Member gets 403 |
| T-12-03 | Tampering / IDOR | A Guardian of family A revoking an invitation in family B by guessing invitationId | high | mitigate | Invitation loaded by Id AND command.FamilyId (re-scoped like RemoveMemberCommand); mismatched family yields a denial, never a cross-family mutation |
| T-12-04 | Information Disclosure | A revoked invite still being redeemable | medium | mitigate | Revoke sets Status=Revoked; RedeemInviteCommand already rejects any non-Pending status |
| T-12-05 | Denial of Service / Integrity | Same-family rejoin after removal violating the unique (FamilyId, UserId) index → unhandled 500 + a consumed (Accepted) but membership-less invite | medium | mitigate | The Accept branch reactivates the existing inactive row in place instead of re-inserting; the invitation Status flip and the reactivation commit in one SaveChangesAsync, so no half-applied state and no unique-index exception (review finding #6) |
</threat_model>
