# Phase 1 — UI Design Contract

**Scope:** Onboarding + Family Circle setup only — the 7 screens in the design system's "01 — Onboarding → Family Setup" feature set (Welcome, Register, Role selection, Create circle, Invite member, Accept/reject, Manage permissions), plus two screens the mockup does NOT cover but Phase 1 requirements need (Login, Forgot/Reset Password — specified below by extension from the same design tokens, since no hifi frame exists for them). ## Scope Resolutions (flagged conflicts from the orchestrator, resolved here)

### 1. Invite mechanism — FAM-02 ("invite by email") vs. mockup (share-code/QR)

**Finding:** `REQUIREMENTS.md` FAM-02 says "User can invite a family member by email." The actual hifi mockup (frame F1-5, "Invite member") shows no email-address-entry field at all — it shows a QR code block, a mono share code (`SP-4K9X`, "Expires in 24h"), a "Copy link" button, a "Share" button (native share-sheet icon `ios_share`), and a "Pending" list keyed by name + relative time ("Jordan (Teen)" / "Invited 2h ago"), not by email address.

**Resolution:** Phase 1 UI implements the mockup's share-code/QR flow as designed — this is the higher-fidelity, more recently-authored source per `SYSTEM_DESIGN (1).md`'s own tie-breaking rule ("this README wins" — and by extension its authored screens win over a one-line requirement gloss). There is **no email-address input field** on the Invite screen. "Invite by email" (FAM-02) is satisfied at the transport layer, not the UI layer: the native "Share" button's share-sheet lets the inviting user send the code/link via email (or SMS, messaging apps, etc.) among other channels — this is how the mockup's own "Share" affordance covers the email case without a dedicated form.
**Executor/planner note:** the backend data model should key invitations by a generated code (with a 24h expiry), not by a required recipient email address. If the planner wants a literal "send an invite email programmatically" as a belt-and-braces channel in addition to the share-sheet, that is a backend/notification decision, not a UI one — it does not require a new screen or field per this contract.

### 2. Persistent app-shell chrome (bottom nav) — is it needed in Phase 1?

**Finding:** The design system's bottom nav (Map / Activity / **SOS** / Insights / Privacy, raised center SOS button) appears on "primary" screens starting at Flow 02 (F2-1, "Home / Live Map"). All 7 Phase 1 screens belong to Flow 01 (Onboarding → Family Setup) and none of them render the bottom nav in the mockup — Welcome has no chrome, Register/Invite/Permissions use a simple back-arrow (+ optional title/"Done") header, Role selection/Create circle use a step-progress header ("STEP 1/3" etc.), and Accept/reject has no header at all (full-bleed content).

**Resolution:** **Non-issue — no bottom-nav shell in Phase 1.** None of the 7 in-scope screens need it, and building it now would be premature (SOS isn't implemented until Phase 3; DESIGN-02's exact SOS button spec is Phase 3's concern, not Phase 1's).

**New gap this surfaces (not in the original two flags, but directly follows from them):** after a user finishes onboarding (Guardian creates a circle and reaches "Manage permissions" with nothing left to configure, or a Member accepts an invite), there is no designed "landing" screen to route to — Home/Live Map is Phase 2/3. **Decision:** Phase 1 must ship a minimal, deliberately undesigned **authenticated landing stub** (not one of the 7 spec'd screens, not styled with the full design system) so the app has somewhere to go post-auth: plain `AppBar` with the circle name, a plain `ListView` of circle members (name + role chip), and a `Logout` action in the app bar overflow menu. Use default Material 3 styling for this stub, **not** the SafePath tokens below — this keeps it visually obviously-a-placeholder so nobody mistakes it for the real Home/Live Map screen Phase 3 will build, and avoids the executor investing design effort in a screen this phase doesn't own. Do not add a bottom nav to this stub.

## Spacing Scale

This project has a **pre-existing, locked design system** (`DESIGN-01` requires exact fidelity), so the values below are read directly from `SYSTEM_DESIGN (1).md`, not the generic 8-point default. Base unit is 4px.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-label gaps, tightest inline spacing |
| sm | 8px | Compact element spacing (e.g. trust-chip icon gap, toggle row internal gaps) |
| xs-md | 12px | Card internal row gaps (e.g. permission-row icon-to-text gap) |
| md | 16px | Default element spacing, field-to-field gaps, standard button vertical padding |
| lg | 24px | Section padding, gap between heading and first control group |
| xl | 32px | Larger layout gaps (used sparingly; most screens cap out at 24-28px) |

**Documented exceptions (from the design system itself, not ad hoc):**
- Screen horizontal padding (gutters): **22–28px** — every screen's content column pads at 22, 24, 26, or 28px depending on frame; use 24px as the phase default unless a specific screen in this contract says otherwise (Welcome uses 28-36px center-aligned padding; Register/Invite/Permissions use 26-28px; Role selection/Create circle use 24px).
- Card internal padding: **14–22px** (e.g. invite-code panel uses 22px, permission toggle rows use 15-18px vertical).
- Minimum tap target: buttons in this contract are 17px vertical padding on a full-width container (≈50-54px total height) — well above the 44px accessibility floor. Toggle switches are 46×28px. No icon-only touch targets below 44px appear in Phase 1 scope.

## Typography

Manrope + JetBrains Mono, per the locked design system. **This system deliberately uses 4 weights (500/600/700/800) plus a 5th (600 mono)** — more than a greenfield "2 weights max" default would call for. This is intentional and required by `DESIGN-01` (exact recreation of an existing, already-shipped-looking 36-screen system), not a fidelity gap — do not collapse it to 2 weights.

| Role | Size | Weight | Line Height | Notes |
|------|------|--------|-------------|-------|
| Display | 38px | 800 (Manrope) | 1.05, letter-spacing -0.025em | Welcome screen wordmark only ("SafePath AI"). Not reused elsewhere in Phase 1. |
| Heading | 25–28px | 800 (Manrope) | 1.15–1.2, letter-spacing -0.02em | Screen-level headlines: "Create account", "Who are you in this circle?", "Name your circle", "Add to The Rivera Family", "Maya invited you to The Rivera Family" |
| Title | 16–17px | 700 (Manrope) | 1.2 | Header bar titles ("Invite", "Permissions"), role-card titles ("Guardian / Parent"), member-row names ("Jordan Rivera") |
| Body | 14–16px | 600 (Manrope) | 1.4–1.5 | Button labels (700 for primary CTAs specifically — see below), input field text (15px/600), permission-row labels (14.5px/700 — treat as Title-weight at Body-size) |
| Body secondary | 12–14px | 500 (Manrope) | 1.4–1.5, color `#5E726F` | Subtitles under headings ("Your circle starts with you."), helper/reassurance text, role-card descriptions |
| Caption / mono label | 11–12px | 600 (JetBrains Mono) | 1.3, letter-spacing 0.04–0.08em, uppercase, color `#8A9893` | Field labels ("FULL NAME", "EMAIL"), step indicator ("STEP 1 / 3"), status badges ("PENDING", "ACTIVE") |
| Code display | 24px | 800 (JetBrains Mono) | 1.0, letter-spacing 0.18em | The invite share code only (`SP-4K9X`). Note: `SYSTEM_DESIGN (1).md`'s font table lists mono weights as "400/500/600" but the mockup itself renders this at 800 — Google Fonts' JetBrains Mono does ship an 800/ExtraBold cut, so this is buildable; flagging only because the README under-documents its own mockup here. |

**Primary CTA buttons specifically** render at **700 weight, 16px, white or brand-colored text**, centered, on a full-width rounded container — this is a distinct, consistent pattern across every screen in scope (Continue / Create circle / Accept & join / Copy link / Share).

## Color

Dominant/secondary/accent below reflect actual usage across the 7 in-scope screens (Welcome's dark-teal-gradient hero is the one deliberate exception, noted below).

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#ECF0EF` (App BG) | Default screen background for Register, Role selection, Create circle, Invite, Accept/reject, Permissions. **Exception:** Welcome only uses a deep-teal gradient (`#1FA89B` → `#0C3A3F`) full-bleed hero background instead — this is the mockup's one intentional departure and should be built as such, not "fixed" to the app-bg default. |
| Secondary (30%) | `#FFFFFF` (Surface) | Cards, input fields, the invite QR/code panel, permission-toggle-row containers, role-selection cards |
| Accent (10%) | `#15807C` (Primary teal) | **Reserved for:** primary CTA button fills (Continue, Create circle, Accept & join, Copy link), focused-input border + 4px focus ring (`rgba(21,128,124,.1)`), selected role-card border + shadow, selected role-card radio-checked icon, step-progress bar fill, active toggle-ON track (share live location / view history / SOS responder), "Done" link text, circle-avatar tile fill (`diversity_3` icon bg), share-code panel's shield icon, secondary outlined button text+border ("Share" button on Invite screen) |
| Accent (secondary, dark-surface only) | `#5FD0C5` (Accent Mint) | **Reserved for:** the Welcome screen's primary CTA fill only ("Create your circle") — used in place of `#15807C` there specifically because it sits on the deep-teal gradient where mint has contrast and teal-on-teal would not. Do not use mint as a general accent substitute anywhere else in Phase 1. |
| Destructive | `#C42A30` (SOS Red — deep variant) | **"Remove from circle" text + icon only** (`person_remove`), on the Manage Permissions screen. **Flagged exception, resolved below.** |

**Flagged color-rule tension (resolved):** `SYSTEM_DESIGN (1).md`'s own critical rule states *"Red (`#DE3B40`) is reserved exclusively for SOS/emergency states... do not dilute red anywhere else."* Yet the hifi mockup itself renders "Remove from circle" in `#C42A30` (an SOS-red-deep variant), which is the only other red usage in the entire 36-screen system. Resolution: **honor the mockup as written** — this is a single, deliberate, confirmation-gated destructive action (not an ambient/routine warning, which is exactly what the rule is protecting against), so it does not "dilute" the SOS meaning the way a routine amber-should-be-red warning would. Constraints on this exception, to keep it from spreading: (1) this hex may be used **only** for the "Remove from circle" label/icon in Phase 1 — no other error, warning, or destructive copy anywhere in this phase may use any red-family color; (2) tapping it must open a confirmation dialog (see Copywriting Contract) before anything destructive happens — it must never fire on a single, un-confirmed tap, to avoid any behavioral (not just visual) confusion with SOS.

**Form validation / error states (not in the mockup — defined here by extension, Claude's discretion, default applied):** the 7 hifi screens show only happy-path states; Login/Register/Forgot-password error states (wrong password, invalid email, expired reset link, expired invite code) are not designed. Per the same red-reservation discipline above, these use **Caution amber** (`#C98A2B` text / `#FBF3E3` bg / `#EFDFBF` border), consistent with the system's own stated rule that non-SOS attention states are amber, not red:
- Invalid input border: 1.5px `#C98A2B`, no glow ring (glow ring is reserved for the focused/valid teal state)
- Inline error copy: 12px/500 Manrope, color `#8A6118`, placed directly under the offending field
- Full-screen error banners (expired link/code): amber card, same visual language as the Caution bg/border/text tokens above

## Screens Not in the Mockup — Specified Here by Extension

`SYSTEM_DESIGN (1).md`'s 36 screens do not include Login or Forgot/Reset Password, even though AUTH-02/AUTH-03/AUTH-04 require them in Phase 1. Built using the same tokens as Register (identical shell: `#ECF0EF` bg, back-arrow header, Manrope 800/28px heading, 500/14px subtitle, field style exactly matching Register's FULL NAME/EMAIL/PASSWORD inputs, full-width 700/16px teal CTA):

- **Login:** Heading "Welcome back." Subtitle "Good to see you again." Fields: EMAIL, PASSWORD (same visual spec as Register). Primary CTA: "Log in". Secondary link (12.5px/500, `#5E726F`, centered under CTA): "Forgot password?". Tertiary link: "Don't have an account? Create one".
- **Forgot password (request):** Heading "Reset your password." Subtitle "We'll email you a link to get back in." Field: EMAIL only. Primary CTA: "Send reset link".
- **Forgot password (confirmation):** Same shell, icon swapped to `mail` at 44px in the `#15807C` accent tile (matching the Create-circle avatar-tile pattern). Heading "Check your email." Subtitle "If an account exists for that address, we've sent a link to reset your password. It expires in 24 hours." No further CTA beyond a "Back to login" text link.
- **Logout:** not a screen — an action (see the authenticated landing stub, Scope Resolutions #2) with a simple confirm dialog: "Log out of SafePath AI?" / Cancel / "Log out".
