<review_findings_addressed>
- HIGH "transfer ownership and delete family are absent" — added here as Guardian-only commands + endpoints.
- MEDIUM "No FKs/cascades/RLS in the family migration" — add FKs from FamilyMembers.FamilyId and FamilyInvitations.FamilyId to Families.Id with cascade delete via a new EF migration + entity-config relationships. RLS decision: deliberately NOT relied upon as primary enforcement (RESEARCH Pitfall 5 — the backend connects with a trusted service-role connection string that bypasses RLS), with server-side FamilyAuthorizationService remaining authoritative. Documented in an ADR so the rationale is visible rather than implicit.
- MEDIUM "FK migration can fail on orphaned rows in the live Supabase DB" (review finding #10) — referential integrity was app-only until now, so the live database may already hold FamilyMembers/FamilyInvitations rows whose FamilyId points at a deleted/absent Family. `dotnet ef database update` would fail on AddForeignKey when such orphans exist. Fixed in Task 1 by hand-prepending scoped orphan-cleanup DELETEs to the generated migration Up, BEFORE the AddForeignKey calls. The DELETEs are scoped strictly to genuinely-orphaned rows (FamilyId NOT IN (SELECT Id FROM Families)) so no live row is ever removed.
- LOW "Transfer ownership doesn't specify Permissions handling" (review finding #14) — the role swap alone would leave the ex-Guardian with FullLocation (granted at create) and the new Guardian with ViewOnly (granted at join). Decision (Claude's discretion — no CONTEXT.md locked decision): permissions SWAP WITH roles. On transfer the new Guardian's Permissions become FullLocation (the CreateFamilyCommand Guardian default) and the ex-Guardian's Permissions become ViewOnly (the RedeemInviteCommand member default), keeping role and permission-level coherent. Asserted in TransferOwnershipCommandTests.
</review_findings_addressed>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Guardian app -> /families lifecycle endpoints | Transfer/delete are re-authorized server-side against the caller's own Guardian membership |
| Client-supplied familyId / memberId | Never trusted; every handler re-scopes target rows to the authorized family |
| Database referential integrity | Enforced by FK/cascade constraints, not only by application code |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-13-01 | Elevation of Privilege | A non-Guardian transferring ownership or deleting the family | high | mitigate | Both handlers call RequireRole(Guardian); a Member gets 403 |
| T-13-02 | Tampering / IDOR | A Guardian of family A deleting or transferring in family B by guessing familyId/memberId | high | mitigate | RequireRole is scoped to command.FamilyId and target rows are re-loaded by Id AND FamilyId; cross-family operations are denied |
| T-13-03 | Denial of Service / Integrity | Deleting a family orphaning member/invitation rows | medium | mitigate | Handler explicitly removes child rows AND FK cascade delete removes them at the DB level (defense-in-depth) |
| T-13-04 | Repudiation | Ownership silently stranded (no Guardian) after transfer | low | mitigate | Transfer atomically promotes the target to Guardian (with FullLocation) while demoting the caller (to Member/ViewOnly) in one SaveChanges; self-transfer is rejected so a distinct owner always exists |
| T-13-05 | Tampering / Integrity | Orphan-cleanup DELETE in the FK migration over-deleting live rows | medium | mitigate | The migration DELETEs are scoped strictly to `FamilyId NOT IN (SELECT Id FROM Families)`, so only rows whose parent Family genuinely does not exist are removed; rows belonging to any existing family are untouched. Verified by an acceptance criterion asserting the exact NOT IN predicate |
</threat_model>
