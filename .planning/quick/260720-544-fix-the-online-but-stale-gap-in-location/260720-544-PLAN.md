<objective>
Close the "online-but-stale" gap on the Live Map. Today online/offline is derived purely from SignalR socket connection state (LocationHub OnConnected/OnDisconnected -> PresenceTracker), so a stationary member with the app open and socket connected shows "online" indefinitely even though their last GPS/battery reading may be many minutes old (geolocator `distanceFilter: 10` never emits while stationary).

Add a client-side "stale" concept that distinguishes connected-but-stale from connected-and-fresh by comparing each member's last-seen timestamp against a staleness threshold, re-evaluated on a periodic timer (Flutter widgets do not re-render from wall-clock time alone), and surface it distinctly in the map marker and member detail sheet.

Purpose: Guardians must be able to tell "I am seeing live, fresh data for this person" from "their app is connected but I have not had a fresh reading in a while" — a core trust property of a family-safety live map.
Output: `isMemberStale(userId)` derived state, a shared staleness threshold + helper, a periodic re-evaluation ticker on the Live Map, and neutral (non-SOS, non-caution) stale styling on the marker presence label, the member status rail, and the detail-sheet badge.
</objective>

  <behavior>
    isStaleAge(age):
    - age below kStaleThreshold (3 min) -> false (fresh)
    - age at exactly kStaleThreshold -> true (stale)
    - age above kStaleThreshold -> true
    - negative age (clock skew / future timestamp) -> false (clamped, treated as fresh)
    LocationState.isMemberStale(userId, {now}):
    - member online AND last-seen age >= kStaleThreshold -> true
    - member online AND last-seen age < kStaleThreshold -> false
    - member offline (regardless of data age) -> false (offline already communicates the gap)
    - member online but memberLastSeenAt is null -> false (no evidence of staleness; a fresh connection carries a recent presence timestamp)
  </behavior>
  <action>
Add a shared staleness threshold and a pure boolean helper to `staleness.dart`, then derive per-member staleness on `LocationState`.

In `staleness.dart`:
- Add `const kStaleThreshold = Duration(minutes: 3);` with a doc comment explaining the choice: it sits just above the existing 2-minute first opacity-fade band in `stalenessFor` (so the marker begins to visually fade slightly before the explicit "stale" label appears — a coherent progression) and is a reasonable "we should have heard something by now" window given the 5-minute movement-independent battery-refresh cadence in LocationController. Do not remove or alter the existing `stalenessFor` opacity bands — the new binary flag is complementary to the continuous opacity fade.
- Add `bool isStaleAge(Duration age, {Duration threshold = kStaleThreshold})` that clamps negative ages to `Duration.zero` (mirroring `stalenessFor`'s existing `age.isNegative ? Duration.zero : age` normalization) and returns `normalized >= threshold`.

In `location_controller.dart`:
- Import `staleness.dart` (same `application/` folder: `import 'staleness.dart';`).
- Add `bool isMemberStale(String userId, {DateTime? now})` to `LocationState`, placed next to the existing `isMemberOnline` / `memberLastSeenAt` methods. It returns `false` unless `isMemberOnline(userId)` is true, then reads `memberLastSeenAt(userId)`; if that is null it returns `false`, otherwise it computes `(now ?? Follow the existing optional-`now` convention used by `lastSeenText` / the test suite so tests can inject a deterministic clock.
- Do not change `isMemberOnline`, `memberLastSeenAt`, or any copyWith/flag behavior — stale is strictly a derived read, no new stored state.

In `staleness_test.dart`:
- Add a `group('isStaleAge', ...)` covering: just below kStaleThreshold is fresh, exactly at kStaleThreshold is stale, above is stale, and a negative duration is treated as fresh.

In `member_presence_test.dart`:
